# 00E-前端二开改造总方案（先吃透原前端，再按图纸升级）

> 勘误说明（2026-03-03）：
> 本方案初版主要基于 `console/src` 观察；后续已确认 `console/assets` 存在更完整的历史 React 控制台能力。
> 以 `00F-原前端不一致勘误与基线修正.md` 为最终基线修正依据。

## 1. 执行原则（你这次定的铁律）

1. **原前端是基础，不推翻重写**。
2. **先盘点原能力，再做二开改造**（先“保命可用”，再“美化增强”）。
3. **每个新页面必须能映射到已有后端 API**，避免空壳 UI。
4. **按 P0→P1→P2 逐层推进**，每轮都可运行、可验证。

---

## 2. 原前端现状（代码实测）

### 2.1 现有结构（console/src）

- 框架：Vue3 + Vite + Naive UI + Vue Router
- 当前主要文件：
  - `main.js`
  - `router/index.js`
  - `layouts/MainLayout.vue`
  - `pages/SimpleTest.vue`
  - `pages/Test.vue`
  - `pages/Home.vue`

### 2.2 现有面板能力（可直接复用）

- 已有布局壳：左侧菜单 + 顶部状态 + 内容区 + 页脚
- 已有路由入口：`/`、`/channels`、`/settings`、`/about`
- 已有马卡龙风格雏形与 Naive UI 基础组件可用

### 2.3 当前短板（为什么必须二开）

- 路由页基本都指向 `SimpleTest`，功能仍是占位
- 缺 API 调用层（暂无独立 `api` 模块）
- 缺复用组件层（`components/` 为空）
- 缺全局错误提示、加载态、统一状态管理

## 5. 二开实施路线（不乱改版本）

## 阶段 A：基础层固化（先稳住）

目标：把“测试页路由”替换成“真实业务页壳”，但不一次塞满功能。

- 建立页面目录：
  - `pages/settings/*`
  - `pages/channels/*`
  - `pages/chat/*`
- 建立 API 目录：
  - `api/config.js`
  - `api/providers.js`
  - `api/envs.js`
  - `api/chats.js`
- 建立基础组件目录：
  - `components/Toast/*`
  - `components/Loading/*`

验收标准：

- 路由可达
- 页面可加载
- API 可打通（至少 GET 列表）

## 阶段 B：P0 核心闭环（先能用）

目标：完成 `76` 文档 P0 条目最小可用闭环。

- 设置页：Provider/Channel/Env 的查看+编辑+保存
- Provider 状态：健康状态、Key 状态、基础统计
- 渠道页：列表、启停、测试按钮
- 全局错误提示：统一 toast + HTTP 错误映射
- 全局加载：页面/按钮/列表 skeleton

验收标准：

- P0 页面可独立操作，不靠手改配置文件
- 主要失败路径有可读错误提示

## 阶段 C：P1 体验增强（再做强）

- Chat 流式体验
- Soul 保护交互提示
- API 管理系统高级功能（排序/统计图）

## 6. 本轮结论（对你的要求的落实）

这次前端改造将按“**原前端为基础 → 按图纸升级二开**”执行，不做推倒重来。

具体落地方式：

1. 先用原项目路由/布局/主题作为骨架。
2. 再按 `75/76/101` 将后端能力映射到页面。
3. 最后按 `63/100/74` 做视觉与交互升级。

这样做的结果是：**功能不断层、实现可验证、不会再陷入错改乱改。**
