# 00F-原前端不一致勘误与基线修正

## 0) 项目定位（治理声明）

你当前的判断作为本阶段共识：

- 原项目质量较差，只能作为历史素材库，不作为实现标准。
- 本阶段工作属于“讨论 + 二开重构”，不是“修修补补旧实现”。
- 说明文档序列（60+）是需求与验收主依据；旧代码与旧产物只用于能力参考与回归对照。

## 1) 勘误结论（你判断是对的）

当前仓库里存在两套前端痕迹，不是同一代代码：

1. `console/src`：Vue3 + Naive UI 的轻量骨架（页面较少，偏占位）
2. `console/assets/index-*.js`：已编译的 **React** 控制台产物（功能明显更完整）

因此，“原来的前端和实际不一样”的根因不是你看错，而是**源码层与打包层不一致**。

---

## 2) 证据摘要（已核对）

### A. Vue 源码骨架

- `console/index.html` 指向 `/src/main.js`
- `console/src/main.js` 是 Vue 启动入口
- `console/src/router/index.js` 路由很少，且多数页面指向测试页

### B. 真实历史能力（React 打包产物）

在 `console/assets/index-*.js` 中可见：

- React 运行时特征（`react-dom` 等）
- 完整路由：`/chat`、`/channels`、`/sessions`、`/cron-jobs`、`/skills`、`/workspace`、`/models`、`/environments`
- 完整 API 客户端调用：`/chats`、`/config/channels`、`/cron/jobs`、`/skills`、`/workspace`、`/models`、`/envs` 等
- 中英文 i18n 文案、页面模块化样式与较完整交互逻辑

这说明“原前端基础”应以 **assets 中这套已成型功能** 为准，而不是仅以 `console/src` 现状判断。

---

## 3) 对二开策略的修正（立即生效）

从现在起，前端二开采用双层基线：

1. **功能基线**：以 `console/assets` 反推出的 React 控制台能力为基准（保功能不丢）
2. **改造载体**：以当前 `console/src`（Vue）作为可维护实现层逐步重建

换句话说：

- 不是“只看 Vue 骨架就重做”；
- 而是“先把 React 产物已有能力清点齐，再在 Vue 里对位复刻与升级”。

---

## 4) 二开实施防错规则（防再跑偏）

1. 每做一页，先对照 `assets` 是否已有同类能力。
2. 每个页面先列“原能力清单”再开始编码。
3. 页面验收必须包含：
   - 路由可达
   - API 打通
   - 与原能力对齐（至少不减）
4. 先恢复核心 8 路由能力，再做 UI 升级与扩展。

---

## 5) 下一步执行（按此修正推进）

优先恢复并对齐以下页面（P0/P1 主干）：

### 执行批次（固定顺序）

#### Wave 1（P0，先保命）

1. `Channels` → 对齐 `/config/channels`
2. `Models` → 对齐 `/models`
3. `Environments` → 对齐 `/envs`

目标：先把“配置中枢”恢复成可用状态。

#### Wave 2（P0/P1，核心运行面）

4. `Chat` → 对齐 `/chats`（含 stream 能力）

6. `Cron Jobs` → 对齐 `/cron/jobs`

目标：把日常运行链路（聊、看会话、跑任务）恢复。

#### Wave 3（P1，运维与扩展）

7. `Skills` → 对齐 `/skills`
8. `Workspace` → 对齐 `/workspace`

目标：恢复技能管理与工作区管理能力。

### 每页统一验收口径（必须全部满足）

1. **路由可达**：页面 URL 可直接访问且无空白页。
2. **列表可读**：至少 1 个 GET 接口成功显示数据或空态。
3. **操作可写**：至少 1 个写操作（PUT/POST/DELETE）成功闭环。
4. **错误可见**：接口失败时有明确错误提示，不静默失败。
5. **加载可感知**：请求期间有 loading/skeleton，不假死。

### 施工约束（防乱改）

- 每次只推进 1 页到“可验收完成”，不并行开 3 页以上。
- 先做 API 对位，再做视觉优化；不允许只做样式不接接口。
- 新增能力必须在文档里标注“原产物无/有”，防止误判为回归问题。
